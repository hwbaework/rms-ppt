## 1. 배경과 목적

에너지 자급자족 플랫폼은 6개 페르소나(발전사업자·수용가·컨설턴트·용역사·SPC·관리자)가
"온보딩·구축 → 발전 개시 → 매칭·계약 → 거래·공급 → 정산·세금 → 이행·증빙"의
엔드투엔드 저니를 함께 수행하는 구조다.

이번 정리의 목적:

1. 페르소나별 유저저니와 실제 메뉴(GNB/LNB)의 불일치를 찾는다
2. 중복·누락을 정리해 "한 업무 = 한 화면"으로 만든다
3. 각 화면이 어느 페르소나의 어느 화면과 연계되는지(데이터 정합 포인트)를 정의한다

수용가를 첫 번째로 정리한다. 이유: 수용가는 플랫폼의 최종 고객이자 매출의 출발점이고,
수용가 화면의 데이터 원천이 대부분 SPC·발전사 화면이라
수용가를 먼저 확정하면 나머지 페르소나의 요구사항이 자동으로 도출되기 때문.

---

## 2. 수용가는 누구이고, 무엇이 제일 궁금한가

**페르소나 정의** — 산업단지 입주 수요기업.
실제 대상: 한국단조 · 무진네트워크 · 성림첨단산업 등 (3차년도 사업계획서 기준).
RE100 무료진단을 입구로 플랫폼에 들어와, 발전원(태양광·연료전지·ORC)과
계약유형(Lease PPA · 직접 PPA · 자가 설치)을 선택해 전력을 구매하고,
요금을 납부하고, RE100/ESG 증빙을 수령한다.

**핵심 관심사 (우선순위 순)**

1. **얼마나 줄었나** — 한전에서 그냥 샀을 때보다 이번 달/누적으로 얼마나 아꼈는가.
   플랫폼에 들어온 의사결정("PPA로 갈아타면 싸진다")이 매달 검증되어야 함.
   경영진 보고용 숫자이기도 하다.
2. **RE100을 얼마나 달성했나** — 달성률·목표 대비 진척.
   특히 수출기업은 고객사 ESG 요구·CBAM 대응 때문에 "증빙을 지금 뽑을 수 있는가"가 실무적으로 급함.
   (무료진단 폼에 '수출 대상국' 항목이 있는 이유)
3. **한전이었다면 얼마나 손해였나** — "한전이었다면 낸 금액" vs "실제 PPA 비용"의 직접 비교.
   1번을 가장 직관적으로 보여주는 형태.
4. 이번 달 요금이 얼마 나올지 — 발전량에 따라 변동되므로 예측 가능성이 불안 요소. 다음 결제일·금액.
5. 부족분 발생 여부 — 계약량을 못 받으면 비싼 한전 요금으로 보충되므로 "부족분 때문에 얼마 더 냈나".

**설계 원칙**: 위 관심사가 대시보드의 위젯 배치 순서가 된다.
기존 대시보드는 발전량 차트가 중심이었는데, 발전량은 발전사의 관심사다.
수용가 대시보드의 주인공은 "절감액·한전 대비 손익"으로 교체한다.

## 3. 수용가 유저저니 (계약 → 결제까지)

| 단계 | 행동 | 화면 | 연계 페르소나 |
|---|---|---|---|
| 1. 진단 | 가입 · RE100 무료진단 신청 | /consulting/diagnosis | → 컨설턴트 (신규 의뢰 수신) |
| 2. 제안 | 제안 검토 → 발전원·계약유형 선택 | /consulting/proposals | ← 컨설턴트 (제안 발송) |
| 3. 계약 | 거래 신청 → 계약 체결 | /ppa/trading → /ppa/contracts | ↔ 발전사 (계약 상대) · SPC (검토) · 관리자 (승인) |
| 4. 사용 | 전력 사용 · 사용량 확인 | /ppa/status · /consumer/usage | ← SPC (매칭) · 발전사 (공급) |
| 5. 정산 | 요금 확인 · 납부 | /ppa/billing/* | ← SPC (정산 확정·세금계산서 발행) |
| 6. 증빙 | RE100/ESG 증빙 수령 | /consumer/re100 · /ppa/documents/* | ← SPC (거래 증빙) · 컨설턴트 (인증 지원) |

**주요 분기**

- 자가발전 가능 여부: 가능 → 자가소비형 태양광 / 불가 → 마켓플레이스에서 제안 받기
- 발전원 선택: 태양광 / 연료전지(24시간 안정·SMP+REC) / ORC(폐열 기반)
- 태양광 계약 3종: Lease PPA(설비 임대·발전량 비례 청구) / 직접 PPA(Onsite·Offsite, 5~20년) / 자가 설치(모니터링만)
- 사용량 vs 계약량: 부족 → 부족분 처리(한전 보충) / 충족 → 정상 정산

## 4. AS-IS — 현재 메뉴 구조와 문제점

### 4.1 현재 구조 (layout.tsx GNB_ITEMS 기준)

```
GNB ① 대시보드 (/consumer)
 ├ 대시보드
 ├ 사용량 분석      /consumer/usage
 ├ 에너지 계약      /consumer/contracts     (170줄)
 ├ 요금 내역        /consumer/billing       (248줄)
 └ RE100/ESG       /consumer/re100         (271줄)
GNB ② 컨설팅 (/consulting)
 ├ 컨설팅 홈 / 무료 진단 / 마켓플레이스 / 받은 제안
GNB ③ 전력거래 (→ /ppa/trading)
 ├ 거래 신청 / 내 계약(1,293줄) / 전력 현황
 ├ 정산·요금: 정산 내역(1,821줄) / 세금계산서 / 사용량 편차

GNB 공통: 탄소거래 / e-Data / VPP / DT
```

### 4.2 문제점

**문제 1 — 같은 업무에 화면이 2벌씩 (핵심)**

| 업무 | 대시보드 LNB (요약판) | 전력거래 LNB (풀기능판) |
|---|---|---|
| 계약 | 에너지 계약 — 계약 목록 테이블뿐 (170줄) | PPA 계약관리 — 프로세스 추적·KPI·갱신/변경/해지 (1,293줄) |
| 요금 | 요금 내역 — 한전 청구 vs Lease 비교·청구서 목록 (248줄) | 정산·요금 — 정산 산출 내역·결제일·미정산 (1,821줄) |
| RE100 | RE100/ESG — 달성률·로드맵 (271줄) | 이행 관리 — 증빙 발급·다운로드 (542줄) |

- 사용자는 "내 계약"을 보러 어디로 가야 할지 모른다
- 두 화면의 숫자가 다르게 계산될 위험 (단일 진실 공급원 부재)
- 유지보수 2배 (rmsppt260605 중복진단의 결론과 동일)

**문제 2 — 저니 순서와 GNB 흐름 불일치**
진단(컨설팅 GNB) → 계약(대시보드? 전력거래?) → 납부(?)로 GNB를 왔다갔다 한다.

**문제 3 — "납부" 단계의 액션 부재**
저니 5단계가 "전기요금 납부"인데 화면은 정산 내역 조회·세금계산서 수취뿐.
청구 확인 → 납부로 이어지는 액션이 없다.

**문제 4 — 대시보드가 발전량 중심**
수용가의 1순위 관심사(절감액·한전 대비)가 대시보드의 주인공이 아니다.
정작 그 콘텐츠("월별 한전 청구 vs Lease" 비교 차트)는 제거 대상인 /consumer/billing에 박혀 있다.

**문제 5 — 명칭 불일치**
LNB 라벨 "사용량 편차" vs 화면 타이틀 "사용량 부족 내역".

**문제 6 — 골격만 있는 화면**
운영 보고서(/ppa/documents/report)는 74줄로 사실상 미완성. 발행 주체(SPC? 관리자?)도 미정의.

## 5. TO-BE — 재구성된 메뉴 구조

### 5.1 전체 구조

```
GNB ① 대시보드 (/consumer) — LNB 없음, 단독 홈
GNB ② 컨설팅
 ├ 무료 진단        /consulting/diagnosis
 ├ 마켓플레이스     /consulting/marketplace
 ├ 받은 제안        /consulting/proposals
 └ 컨설팅 현황      /consulting  (舊 '컨설팅 홈' 명칭 변경)
GNB ③ 전력거래
 ├ 거래 신청        /ppa/trading
 ├ 내 계약          /ppa/contracts          ← '에너지 계약' 요약판 통합
 ├ 전력 현황        /ppa/status              (실시간)
 ├ 사용량 분석      /consumer/usage          ← 대시보드 LNB에서 이동 (과거 패턴)
 ├ 정산·요금
 │  ├ 정산 내역     /ppa/billing/settlement  ← 납부 통합 + 한전 비교 상세
 │  ├ 세금계산서    /ppa/billing/tax-invoice
 │  └ 사용량 편차   /ppa/billing/usage-deviation  (명칭 통일)
 └ RE100·이행
    ├ RE100 현황    /consumer/re100          ← 대시보드 LNB에서 이동
    ├ 이행증빙      /ppa/documents/evidence
    └ 운영 보고서   /ppa/documents/report
GNB 공통: 탄소거래 / e-Data / VPP / DT  (노출 유지 여부는 별도 논점)
```

### 5.2 재구성 원칙

1. **한 업무 = 한 화면** — 요약판 3개(에너지 계약·요금 내역)는 LNB에서 제거,
   유일 콘텐츠(한전 비교 차트)는 대시보드와 정산 내역으로 이관.
   RE100/ESG는 화면 유지하되 전력거래 > RE100·이행으로 위치 이동.
2. **저니 순서 = 메뉴 순서** — 컨설팅(진단·제안) → 전력거래(계약→사용→정산→증빙).
   GNB ③ 안에서 LNB 순서도 저니 순서를 따른다.
3. **대시보드는 관심사 순** — "얼마나 아꼈나"가 첫 화면 첫 위젯.
   LNB 2개짜리 GNB를 없애고 대시보드를 단독 홈으로(발전사 /dashboard와 동일 패턴).
   모든 위젯이 상세 화면으로 딥링크.
4. **사용량 3화면의 역할 분리** — 전력 현황(지금·실시간 매칭) /
   사용량 분석(과거·패턴 탐색) / 사용량 편차(부족분·정산 영향)를 한 GNB에 모음.
5. **구조만 정리** — 화면 내부는 건드리지 않음. 콘텐츠 이관·신설이 필요한 지점은
   별도 표시(§8)하고, 필요 시 내용을 조사해 채운다.

## 6. 대시보드 상세 정의 (/consumer, LNB 없음)

관심사 우선순위가 그대로 Zone 순서가 된다.

### Zone 1 — 얼마나 아꼈나 (히어로 영역)

- KPI 3개: **이번 달 절감액** / **누적 절감액(YTD)** / **절감률(%)**
- 메인 차트: **"한전이었다면" vs "실제 PPA 비용" 월별 비교 (12개월)**
  - 舊 /consumer/billing의 '월별 한전 청구 vs Lease' 차트를 여기로 승격
  - 두 선 사이 차이 영역을 음영 처리 → "한전 그대로였으면 +XXX만원 더 냈습니다" 한 문장 요약
- 딥링크: 정산 내역(상세 비교)

### Zone 2 — RE100 얼마나 달성했나

- KPI: **RE100 달성률**(목표 대비 게이지) / 이번 달 RE 비율 / CO₂ 저감
- 액션 버튼: **"증빙 발급"** → 전력거래 > RE100·이행 > 이행증빙 직행
  (수출기업의 "증빙 지금 뽑기" 요구를 한 클릭으로)
- 딥링크: RE100 현황

### Zone 3 — 이번 달 돈 관리

- 이번 달 예상 요금 / **다음 결제일** / 미정산 건수
- **부족분 경고** (조건부 노출): 발생 시에만
  "부족분 XXkWh → 한전 단가 보충 +XX원" → 사용량 편차 딥링크
- 딥링크: 정산 내역

### Zone 4 — 사용량 요약 (미니)

- 이번 달 총 사용량 / 수요 피크 + 월별 미니 차트
- 딥링크: 사용량 분석

### 제거 항목

- 발전량 추이 차트, 발전소 상태 보드 → 발전사 관심사. 재생에너지 발전량은 Zone 2에 흡수.

## 7. GNB별 화면 상세 정의 + 페르소나 연계

### 7.1 컨설팅 GNB (저니 1~2단계)

#### 무료 진단 /consulting/diagnosis (635줄)

5단계 위저드 폼:

1. 컨설팅 분야 선택
2. 기업 현황 — 직원 수 · 연매출(억원) · 연간 가스 사용량(TJ) · 온실가스 배출량(tCO2eq) · 사업장 수 · 사업장 지역
3. 재생에너지 현황·목표 — 수출 대상국 · Scope 1/2/3 배출 · 옥상/부지 가용면적(m²) · 최대 수요(kW) · 월 피크요금(원)
4. 연락처 — 담당자 · 회사명 · 이메일 · 연락처
5. 입력 확인 · 제출

**연계**: 제출 → **컨설턴트**의 신규 의뢰 수신함(/consulting/consulting-requests)으로 핸드오프.
저니 전체의 시작점.

#### 마켓플레이스 /consulting/marketplace (231줄)

- 컨설턴트 카드 목록 탐색·선택
- **연계**: 컨설턴트 프로필(/consultant/profile)이 노출 콘텐츠의 원천.
  자가발전 불가 판정 시 진입하는 분기 경로.

#### 받은 제안 /consulting/proposals (250줄)

- 제안 목록 + 제안 상세 (발전원·계약유형 검토)
- **연계**: ← **컨설턴트**의 제안 관리(/consultant/proposals)에서 발송된 제안의 수신측.
  수락 → 전력거래 > 거래 신청으로 이어지는 다리.

#### 컨설팅 현황 /consulting (472줄, 舊 컨설팅 홈 — 명칭 변경)

- 서비스 분야 소개 / 진행 중인 컨설팅 추적 / 다가오는 일정
- **연계**: ← **컨설턴트**의 고객·프로젝트 관리(/consultant/clients)와 같은 프로젝트 데이터의 수용가측 뷰.
- 비고: 대시보드와 역할이 겹치는 소개성 콘텐츠는 축소하고 "진행 중 컨설팅 추적"을 중심으로.

### 7.2 전력거래 GNB (저니 3~6단계)

#### 거래 신청 /ppa/trading (941줄)

**화면 내용**

#### 내 계약 /ppa/contracts (1,293줄)

**화면 내용**

- KPI: 총 계약 · 이달 총 비용 · CFE 매칭 · RE100 · 자원 믹스
- 계약 상세(사업장·PPA 단위 필터) + 액션: 갱신 신청 / 변경 신청 / 해지 신청 / 신청 취소

**페르소나 연계**

- ↔ **발전사**: 같은 계약의 양 당사자 — 발전사측 화면은 /generator/ppa/contracts.
  **두 화면의 계약 상태가 항상 동일해야 함 (단일 계약 데이터)**
- → **SPC**: 'SPC 검토' 단계 수행 (/platform/ppa/dashboard 계약 현황)
- → **관리자**: 승인 단계 (/platform/trading)
- 갱신/변경/해지 신청 → SPC·관리자의 처리 태스크로 전달 필요

#### 전력 현황 /ppa/status (556줄)

**화면 내용**

- 현재 시각 자원별 매칭 현황(실시간) / 할 일 / 최근 활동

**페르소나 연계**

- ← **SPC**: 매칭 결과의 수용가측 뷰 — SPC 시간단위 매칭(/platform/ppa/status/matching)이 원천.
  **SPC가 보는 매칭과 수용가가 보는 매칭은 같은 데이터여야 함**
- ← **발전사**: 자원별 공급량 = 발전사 발전 현황(/generator/ppa/dashboard)의 수요측 거울

#### 사용량 분석 /consumer/usage (226줄) — 대시보드에서 이동

**화면 내용**

- KPI: 이번 달 총 사용량 · 한전 사용량 · 평균 일사용량 · 수요 피크
- 차트: 시간대별 / 일별 / 월별 사용량

**페르소나 연계**

- → **SPC**: 사용 패턴이 발전량 예측·매칭 알고리즘(/platform/ppa/forecast)의 입력.
  수요 피크가 매칭 우선순위에 반영
- → **컨설턴트**: 진단·제안 시 참조하는 기초 데이터 (무료진단 입력 항목과 연결)

**위치 이동의 근거**: 대시보드(모니터링·10초 확인)와 분석(탐색 도구)은 역할이 다르고,
전력 현황(지금)–사용량 분석(과거)–사용량 편차(부족분)를 한 GNB에 모으면
사용량 관련 3화면의 경계가 명확해진다. LNB 2개짜리 대시보드 GNB도 해소.

#### 정산·요금 > 정산 내역 /ppa/billing/settlement (1,821줄 — 최대 화면)

**화면 내용 (현재)**

- KPI: 이달 총 비용 · 이달 순절감 · 다음 결제일 · 누적(YTD) · 미정산
- 계약 정보: 계약 기간 · 주소 · 설비코드 · 설비용량 · 거래기관 · 계약번호 · 총 용량 · 발전소 수 · 단가 · 상업운전시작일
- 정산 산출 내역: 발전량(계량기/RTU) × SMP 단가 → 공급가액, 발전일·발전시간, 24/7 매칭률

**추가할 것 (확정사항)**

- **납부 상태·납부 액션 통합** — 별도 '청구·납부' 화면 신설하지 않음
- **한전 비교 상세** — 舊 /consumer/billing의 '월별 한전 청구 vs Lease' 차트의 상세 버전 (대시보드 Zone 1과 동일 데이터)

**페르소나 연계**

- ← **SPC**: SPC 통합 정산(/platform/ppa/billing/settlement)에서 잠정→확정된 결과가 내려옴.
  **SPC 확정 전에는 수용가 화면에 '잠정' 상태 표시 필요**
- → **SPC**: 수용가의 납부 = SPC의 수금(/platform/ppa/billing/settlement/payment).
  **납부 상태가 양쪽 화면에서 일치해야 함**
- ↔ **발전사**: 수용가가 내는 돈 ↔ 발전사가 받는 판매대금(/generator/ppa/revenue/invoices).
  사이의 차액 = SPC 마진(/platform/ppa/status/margin, 관리자 전용)
- 분쟁·재정산 제기 → SPC 정산 관리의 분쟁 플로우(잠정/확정/분쟁·재정산)로 연결

#### 정산·요금 > 세금계산서 /ppa/billing/tax-invoice (661줄)

**화면 내용**

- 수취 현황 KPI: 총 발급건수 · 총 수취 합계 · 미발급 · 결제 대기
- 계산서 상세: 발급번호 · 발급일 · 유형 · 공급자 · 사업자번호 · 발전소 · 공급량 · 단가 · 공급가액 · 부가세(10%) · 결제일 · 상태 · 가상계좌

**페르소나 연계**

- ← **SPC**: 발행 주체(/platform/ppa/billing/tax-invoice) — 수용가는 수취측.
  정산 확정 → 발행 → 수취 순서가 강제되어야 함

**역할 분리 (중요)**: 결제 대기·가상계좌 정보가 이미 이 화면에 있어
정산 내역의 납부 기능과 중복될 수 있음 →
**납부 액션은 정산 내역 한 곳**, 세금계산서는 **증빙 문서 보관·수취 확인** 용도로 역할 분리.

#### 정산·요금 > 사용량 편차 /ppa/billing/usage-deviation (806줄)

**화면 내용** (화면 타이틀 "사용량 부족 내역" → LNB 라벨과 통일 필요)

- KPI: 총 부족량(이번달) · 내부 보완률 · 정산 순영향
- 일별 부족량 추이 / 시간대별 부족 분포 / 원인 분해 / 처리 분해 / 정산 영향 합계 / 부족 이벤트 로그
- 액션: 편차 정산 처리

**페르소나 연계**

- ← **SPC**: 부족 판정의 원천 = SPC 예측 vs 실측(/platform/ppa/forecast/error 오차·페널티).
  원인 분해(발전 부족 vs 수요 초과)는 SPC 데이터 없이 계산 불가
- ↔ **발전사**: 발전측 미달이 원인이면 발전사 편차(/generator/ppa/revenue/deviation)와

## 9. 남은 논점

1. **공통 GNB 4개(탄소거래·e-Data·VPP·DT)** — 수용가에게도 전부 노출 중.
   페르소나 제한 없는 iframe성 메뉴인데 수용가 저니와 무관 → 노출 정책 결정 필요
2. **운영 보고서 발행 주체** — SPC vs 관리자, 생성측 화면 미존재
3. **이행증빙 라우트 공유** — 수용가·발전사가 동일 화면 사용, 권한·뷰 분리 설계
4. **다음 순서** — 발전사업자 → 컨설턴트 → 용역사(페르소나 신설 필요) → SPC → 관리자
