## 슬라이드 2 — 수용가는 누구이고 무엇이 궁금한가

**페르소나**: 산업단지 입주 수요기업 (한국단조·무진네트워크·성림첨단산업 등)
RE100 무료진단으로 진입 → 발전원·계약유형 선택 → 전력 구매·납부 → RE100/ESG 증빙 수령

**핵심 관심사 (우선순위)**

1. 얼마나 줄었나 — 한전 대비 이번 달/누적 절감액 (경영진 보고 숫자)
2. RE100 얼마나 달성했나 — 달성률·증빙 (수출기업은 CBAM·ESG 대응으로 시급)
3. 한전이었다면 얼마나 손해였나 — "한전 청구 vs 실제 PPA" 직접 비교
4. 이번 달 요금·다음 결제일 — 발전량 변동으로 예측 불안
5. 부족분 발생 여부 — 부족분은 비싼 한전 요금으로 보충됨

## 슬라이드 3 — AS-IS 문제점

**문제 1. 같은 업무에 화면이 2벌 (중복)**

| 업무 | 대시보드 LNB (요약판) | 전력거래 LNB (풀기능) |
|---|---|---|
| 계약 | 에너지 계약 (170줄) | PPA 계약관리 (1,293줄) |
| 요금 | 요금 내역 (248줄) | 정산·요금 (1,821줄) |
| RE100 | RE100/ESG (271줄) | 이행 관리 (542줄) |

**문제 2. 전력거래 LNB 과적** — 거래+정산+증빙 8~9개가 한 GNB에 섞임
**문제 3. "납부" 액션 부재** — 조회·수취만 있고 결제 액션 없음
**문제 4. 대시보드가 발전량 중심** — 수용가 1순위(절감액)가 주인공 아님
**문제 5. 명칭 불일치** — LNB "사용량 편차" vs 화면 "사용량 부족 내역"
**문제 6. 정산 미러 중복** — /ppa/billing ↔ /lease/billing (복제)

## 슬라이드 4 — 업계 표준 근거 (정산 분리)

검색으로 확인된 IA 패턴:

- **에너지 거래 플랫폼** — Syntphony는 거래를 Clients/Operations/**Back-office(정산)**/Integrations 4모듈로 분리. PowerCore도 contracts/**settlements**/analytics 독립 모듈.
- **ERP** — NetSuite·Oracle 모두 Order Management(주문)와 **Finance/Billing(청구·재무)**을 별도 모듈로. "billing is functionally distinct from sales orders".
- **SaaS 네비게이션 원칙** — ① 조직 논리 아닌 사용자 멘탈 모델로 묶어라(수용가는 "거래"와 "돈 관리"를 다른 일로 인식) ② 최상위 메뉴 5~7개 ③ 결제·요금 같은 핵심 평가 지점은 상위 노출.

→ 결론: 전력거래 안의 정산을 떼어 **독립 GNB로 승격**하는 것이 표준.

## 슬라이드 5 — TO-BE 전체 GNB 구조

```
GNB ① 대시보드     /consumer            LNB 없음, 단독 홈
GNB ② 컨설팅        진단·제안 단계
GNB ③ 전력거래      "거래하기" (계약+사용)
GNB ④ 정산·요금     "내가 내는 돈" (독립 승격)
GNB ⑤ RE100·이행    "증빙" (독립 승격)
GNB 공통           탄소거래 · e-Data · VPP · DT
```

3대 변화:
① 정산·요금을 전력거래에서 분리 → 독립 GNB
② 사용량 분석을 대시보드 → 전력거래로 이동 (현황·분석 한 곳)
③ RE100·이행을 독립 GNB로 승격 (2순위 관심사 상위 노출)

## 슬라이드 6 — 대시보드 (/consumer, LNB 없음)

관심사 우선순위 = 위젯 순서

- **Zone 1 얼마나 아꼈나** — 이번달/누적 절감액·절감률 KPI + "한전 vs 실제 PPA" 월별 비교 차트(차이 음영) → "한전이었으면 +XXX만원"
- **Zone 2 RE100 달성** — 달성률 게이지·RE 비율·CO₂ 저감 + "증빙 발급" 바로가기
- **Zone 3 이번 달 돈 관리** — 예상 요금·다음 결제일·미정산 + 부족분 경고(조건부)
- **Zone 4 사용량 요약** — 총 사용량·수요 피크 + 미니 차트 → 사용량 분석 딥링크
- 제거: 발전량 추이 차트·발전소 상태 보드 (발전사 관심사)

## 슬라이드 8 — GNB ③ 전력거래 (하위 항목)

**거래 신청** `/ppa/trading`
Lease PPA 문의 / 직접 PPA 유형 선택(Onsite·Offsite) / 거래 이력 / 신청 취소
연계: → SPC 매칭·중개 / ↔ 발전사 공급신청 / → 관리자 승인

**내 계약** `/ppa/contracts` (舊 '에너지 계약' 통합)
진행 5단계 추적(신청접수→SPC검토→거래소신고서→승인대기→효력발생)
KPI: 총 계약·이달 비용·CFE 매칭·RE100·자원 믹스 / 갱신·변경·해지
연계: ↔ 발전사 동일 계약(상태 동기화) / → SPC·관리자 검토·승인

**전력 현황** `/ppa/status`
실시간 자원별 매칭 / RE100 누적 이행률·예상 정산금 / 할 일·최근 활동
연계: ← SPC 매칭 결과 / ← 발전사 공급량

**사용량 분석** `/consumer/usage` (대시보드에서 이동)
총 사용량·한전 사용량·평균 일사용량·수요 피크 / 시간대별·일별·월별 차트
연계: → SPC 발전량 예측 입력 / → 컨설턴트 진단 기초자료

## 슬라이드 9 — GNB ④ 정산·요금 (하위 항목)

**정산 내역** `/ppa/billing/settlement` (핵심 화면)
- KPI: 이달 총 비용·이달 순절감·다음 결제일·누적(YTD)·미정산
- 정산 산출: 발전량(계량기/RTU) × SMP 단가 → 공급가액 / 24/7 매칭률
- +납부 통합: 청구 확인 → 납부 액션 (별도 화면 신설 안 함)
- +한전 비교 상세: 대시보드 Zone1의 상세판
- 상태: SPC 확정 전 '잠정' 배지
- 연계: ← SPC 정산 확정 / 납부=SPC 수금 / 발전사 판매대금과 차액=SPC 마진

**세금계산서** `/ppa/billing/tax-invoice`
- 수취 현황: 총 발급건수·총 수취 합계·미발급·결제 대기
- 상세: 발급번호·공급자·사업자번호·공급량·단가·공급가액·부가세·결제일·가상계좌
- 역할: 증빙·수취 확인 전용 (납부 액션은 정산 내역에만)
- 연계: ← SPC 발행 (확정→발행→수취 순서 강제)

**사용량 편차** `/ppa/billing/usage-deviation` (화면명 통일 필요)
- KPI: 총 부족량·내부 보완률·정산 순영향
- 일별 추이·시간대별 분포·원인 분해·처리 분해·이벤트 로그·편차 정산 처리
- 연계: ← SPC 예측 vs 실측 / ↔ 발전사 편차(동일 이벤트 양면)

## 슬라이드 10 — GNB ⑤ RE100·이행 (하위 항목)

**RE100 현황** `/consumer/re100` (대시보드에서 이동)
현재 RE 비율·재생E 사용·CO₂ 저감 / 월별 RE 비율·연도별 목표 vs 실적·에너지원 구성·로드맵
연계: ← 컨설턴트 RE100 인증 지원

**이행증빙** `/ppa/documents/evidence`
Lease 누적 발전·리스료·절감·증빙 제출 / 증빙 문서 + 미리보기·다운로드·이메일·공유
연계: ← SPC 거래 증빙 / ← 발전사 REC 발급(동일 라우트, 권한 분리 필요)

**운영 보고서** `/ppa/documents/report` (현재 74줄, 보강 대상)
누적 발전량·리스료·한전 대비 절감·연간 추정 / 월별 비용 vs 절감 / 발급 이력
연계: 발행 주체(SPC/관리자) 미정의 → 보강 필요

## 슬라이드 11 — 정산 데이터 흐름 (단일 데이터, 다중 뷰)

```
발전사 발전량(계량기/RTU)
        │
        ▼
SPC 정산  ① 잠정 → ② 확정 → ③ 세금계산서 → ④ 수금/지급
        │  (분쟁 시 재정산·수정세금계산서로 분기)
        ├─→ 수용가 : 정산 내역(잠정/확정) → 납부
        ├─→ 발전사 : 판매대금 = 수용가 지불 − SPC 마진
        └─→ 관리자 : SPC 마진 = 청구액 − 지급액
```

미러 라우트로 복제하지 않고 **같은 데이터를 권한·관점만 다르게** 표시.
정산 4상태(잠정→확정→세금계산서→수금/지급)가 전 페르소나 화면에서 동기화.
